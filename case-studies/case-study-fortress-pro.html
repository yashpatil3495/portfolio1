01 The Problem

WEAK PASSWORDS EVERYWHERE

Core Problem

Over 80% of data breaches involve compromised or weak passwords. Most password generators are either too simple (just random ASCII), too opaque (no feedback on strength), or rely on Math.random() — which is cryptographically insecure.

The goal was to build something that was genuinely cryptographically secure — using the browser's actual CSPRNG — while also being legible to non-technical users through real-time entropy visualization.

Security tools often have terrible UX because their designers assume users don't care about aesthetics. Fortress Pro was an experiment in proving that wrong.

02 Research & Discovery

THE SECURITY-UX GAP

I studied the cryptography behind password security — specifically Shannon entropy (measured in bits), the difference between CSPRNG and PRNG, and how password cracking tools like Hashcat approach brute-force attacks.

🔒
CSPRNG vs PRNG
Math.random() is predictable with enough samples. crypto.getRandomValues() uses OS-level entropy — genuinely random.
📊
Entropy Bits
A 12-char password from a 72-char set has ~74 bits of entropy — considered strong for 2025 threat models.
⏱
Crack Time
Translating entropy into "time to crack" makes security tangible. "128-bit entropy" means nothing. "2.3 trillion years" means something.
🧩
Character Sets
Letting users customize character inclusion/exclusion gives them control without compromising the underlying randomness source.
03 Design Decisions

SECURITY MADE LEGIBLE

01
WEB CRYPTO API EXCLUSIVELY

Every password is generated using crypto.getRandomValues() — the browser's CSPRNG. I explicitly avoided Math.random(). This is non-negotiable for a security tool and was the first architectural decision made before any UI work.

Security Architecture
02
REAL-TIME ENTROPY METER

As the user adjusts length and character sets, a live entropy meter updates in bits with a corresponding "crack time estimate" (e.g., "centuries at 10B guesses/sec"). This teaches users about security without lecturing them.

Data Visualization
03
SESSION HISTORY SYSTEM

Generated passwords are stored in a session history (in-memory, not persisted) so users can compare and go back to a previous one. The history clears on page refresh — intentional ephemeral design for a security context.

UX Pattern
04
ZERO NETWORK REQUESTS

Fortress Pro makes zero outbound requests. No analytics, no API calls, no logging. The password never leaves the browser. This is the only appropriate architecture for a security tool — and a key part of the user trust proposition.

Privacy by Design

06 What I Learned

LESSONS CARRIED FORWARD

✦Security and UX aren't opposites. The best security tools make the right choice the easy choice. Good UI doesn't compromise security — it amplifies it.
✦Teach, don't tell. Showing "74 bits of entropy (strong)" educates users about why length matters without ever mentioning cryptography jargon.
✦Privacy by default is a feature. "Zero network requests" became a selling point, not just an implementation detail. Users feel safer when they understand the architecture.
