01 The Problem

HIDING IN PLAIN SIGHT

Core Concept

Encryption makes messages unreadable. Steganography makes them invisible. The challenge: implement pixel-level LSB steganography entirely in the browser, with a UI so simple anyone can use it — no cryptography background required.

I was fascinated by the idea that every image is a potential carrier for hidden data. A JPEG of a sunset can contain a complete message, and no viewer would ever know. The browser's Canvas API makes this possible without any server-side processing.

The creative challenge was making this feel magical — not technical. The UX had to make the encoding/decoding feel like a conjuring trick, not a software operation.

02 Research & Discovery

THE SCIENCE OF INVISIBILITY

I researched Least Significant Bit (LSB) steganography — a technique where the lowest-order bit of each pixel's color channel (R, G, B) is replaced with a bit of the hidden message. Humans can't perceive the resulting color shift (1/256 of a shade), but the data is perfectly recoverable.

🎨
LSB Encoding
Modifying only the last bit of each R, G, B value produces color changes of less than 0.4% — completely imperceptible to the human eye.
📐
Capacity Math
A 1024×768 image has ~786K pixels × 3 channels = 2.3M bits of hiding capacity. That's ~290KB of text — enough for a novel chapter.
🖥
Canvas API Access
ImageData.data gives direct read/write access to every pixel's RGBA values — this is the foundation the entire project is built on.
⚠️
Format Caveat
JPEG compression destroys LSB data — the output must be PNG (lossless). This became a key UX design constraint: always export as PNG.
03 Design Decisions

MAKING MAGIC USABLE

01
DRAG-AND-DROP FIRST

The first interaction is dragging an image directly onto the canvas. No file picker dialog, no button hunting. This immediately signals that the tool is interactive and direct — you're working with pixels, so the experience should feel hands-on.

Interaction Design
02
ENCODE / DECODE MODE TOGGLE

A single toggle switches between encoding (hiding a message) and decoding (extracting one). The UI restructures around the mode — encode shows a text input, decode shows extracted output. Same image slot, same canvas, different operation.

UX Architecture
03
ALWAYS EXPORT AS PNG

When the user downloads the encoded image, CipherPixel forces PNG format via a canvas.toBlob() call — regardless of the input format. JPEG compression would corrupt the hidden data. This is done transparently: the user just clicks "Download" and gets a PNG.

Technical Constraint → UX Decision
04
LSB DELIMITER PATTERN

The message is terminated with a null byte sequence (8 zero bits) so the decoder knows when to stop reading. Without this, the decoder would try to read every pixel in the image and return garbage after the real message ends.

Algorithm Design
04 Technical Stack

THE CANVAS AS A FILESYSTEM

Canvas APIImageData APIFile APIDrag & Drop APILSB SteganographyBlob APIVanilla JS

The core encode function iterates pixel by pixel, replacing the last bit of R, G, B values with message bits. The decode function reads those same bits back and reconstructs the original character string. All of this happens synchronously on the canvas — no server ever sees the image or the message.

06 What I Learned

LESSONS CARRIED FORWARD

✦Platform primitives are powerful. The Canvas API gives you direct pixel access — that's enough to implement image steganography entirely client-side. Most developers underestimate how much the browser can already do.
✦Format matters deeply. The JPEG/PNG distinction wasn't a minor technical note — it defined a core UX constraint. Technical decisions always have design consequences.
✦Delight is a design goal. CipherPixel isn't solving a critical workflow problem. It's delightful — and delight is a legitimate and underrated design objective. Sometimes the best projects are the ones that make people say "wait, that's possible?"
