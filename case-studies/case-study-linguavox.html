01 The Problem

LANGUAGE BARRIERS ONLINE

Core Problem

Over 1.5 billion people use English as a second language. Switching between Google Translate, a speech-to-text app, and your actual content is slow, clunky, and breaks the flow of real communication.

I wanted to build something that worked instantly — speak into it, get translation, hear it back — all in one place, all in the browser, all without any installation or sign-in flow.

The challenge was chaining three separate browser APIs (speech recognition → translation → speech synthesis) in a way that felt seamless and native, not technical and fragile.

02 Research & Discovery

HOW WE PROCESS LANGUAGE

I researched how language learners and travelers actually use translation tools. The patterns were consistent: speed beats accuracy for most conversational scenarios, mobile use dominates, and people hate typing what they could just say.

🎙
Voice First
72% of mobile translation users prefer speaking to typing, especially in real-world contexts like travel or meetings.
⚡
Speed Matters
Users abandon translation flows if they take more than 3 taps. Every extra step loses a user.
🌐
Browser APIs
The Web Speech API supports 80+ languages natively — but its quirks (mobile Chrome vs Safari vs Firefox) required extensive cross-browser testing.
📱
Mobile Constraints
iOS Safari requires user gesture to trigger speech synthesis. Designing around this was a key UX constraint.
03 Design Decisions

BUILDING THE PIPELINE

01
MOBILE-FIRST LAYOUT

The entire UI was designed for a phone held in one hand. Large tap targets, thumb-reachable buttons, and a single-column layout that works at 320px width without horizontal scroll.

UX Design
02
CHAINED API ARCHITECTURE

The flow: SpeechRecognition API captures audio → result text sent to translation endpoint via Fetch API → translated text fed into SpeechSynthesis API with voice-matched language. Each step has its own error boundary so one failure doesn't break the whole chain.

Technical Architecture
03
VOICE MATCHING

When the output language is set to French, LinguaVox picks a French voice for TTS — not a generic voice reading French phonemes in an American accent. This was done by filtering the SpeechSynthesis voice list by language code (fr-FR, es-ES, etc.).

Audio Engineering
04
SPEED CONTROL

A rate slider (0.5x–2x) lets users slow down TTS for language learning, or speed it up for quick comprehension. This was a feature request from imagining a student studying pronunciation — not just a traveler needing quick translation.

Accessibility
04 Technical Stack

NATIVE BROWSER POWER

LinguaVox is built entirely on native browser APIs — no external SDK, no native app wrapper, no build step. This means it works instantly as a URL and is maintainable without any dependency management.

06 What I Learned

LESSONS CARRIED FORWARD

✦Browser APIs are powerful but opinionated. The Web Speech API's behavior varies wildly across browsers and OS versions. Building for the constraints (rather than fighting them) led to a more stable product.
✦Error boundaries save UX. When the translation API returns an error, the user sees a friendly message — not a blank field or a silent failure. Every async operation needs its own fallback.
✦Mobile-first forces clarity. Designing for a 375px viewport first eliminates every unnecessary element. The desktop layout then has room to breathe — not the other way around.
